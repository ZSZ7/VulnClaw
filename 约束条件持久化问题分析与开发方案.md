## 1. 结论

该 issue 是真实存在的，不是偶发现象。

当前 VulnClaw 在自主渗透和持续性渗透流程中，用户提供的“任务约束条件”没有被结构化保存，也没有被每一轮强制注入 prompt。它们主要只是作为首轮自然语言输入的一部分进入上下文，随后会因为以下机制逐步衰减：

1. 后续 round context 只保留目标、阶段、最近步骤、finding、notes 等摘要，不会重复强调“仅允许测试某端口/某路径/某协议”之类的硬约束。
2. 用户显式提示只会通过 `user_vuln_hint_rounds` 保留 3 轮，且这个机制只覆盖“漏洞类型提示”，并不覆盖通用范围约束。
3. 对话历史会被压缩和裁剪，压缩摘要偏向发现、失败、工具结果，而不是保真地保存任务边界。
4. `persistent_pentest` 从第 2 个 cycle 起，会把原始输入替换成泛化提示“继续对目标进行渗透测试”，原始约束进一步丢失。

因此，用户如果一开始说“只测 443 端口”或“只允许针对 8080 端口做测试”，模型在几轮之后确实可能忘记该限制，并转向其他端口或更广泛的攻击面。

在 [vulnclaw/agent/prompt_context.py](C:/Users/UncleC/Desktop/VulnClaw/vulnclaw/agent/prompt_context.py:8) 的 `build_round_context()` 中，保留的是：

但没有“当前硬性约束”段落。

### 2.4 用户明确提示只保留 3 轮，且不适用于范围约束

在 [vulnclaw/agent/core.py](C:/Users/UncleC/Desktop/VulnClaw/vulnclaw/agent/core.py:103)：

- `user_vuln_hint_rounds = 3 if self.runtime.user_vuln_hint else 0`

在 [vulnclaw/agent/prompt_context.py](C:/Users/UncleC/Desktop/VulnClaw/vulnclaw/agent/prompt_context.py:18)：

- 只有 `user_vuln_hint` 会按“第 N/剩余轮次”方式注入
- 注入一次后还会 `agent.runtime.user_vuln_hint_rounds -= 1`

这意味着它天然是衰减型机制，而且仅适合“这个点测 SQL 注入/XSS”这种提示，不适合“只允许端口 443”这种硬边界。

### 2.5 历史消息会被压缩和强裁剪

在 [vulnclaw/agent/context.py](C:/Users/UncleC/Desktop/VulnClaw/vulnclaw/agent/context.py:717)：

- 超过 `max_history` 后会进入 `_trim()`
- 旧消息会被 `_compress_messages(old)` 压缩成 `[之前的会话摘要]`

而 `_compress_messages()` 在 [vulnclaw/agent/context.py](C:/Users/UncleC/Desktop/VulnClaw/vulnclaw/agent/context.py:751) 主要提取：

- 工具调用结果
- 发现/漏洞/端口/路径/失败/验证成功等关键词

它并没有把“只测试某端口”“不要扫描其他端口”视为必须保真的约束对象。

此外在 [vulnclaw/agent/context.py](C:/Users/UncleC/Desktop/VulnClaw/vulnclaw/agent/context.py:784) 还有：

- `trim_messages(max_messages=20)` 直接保留最后 20 条消息

一旦发生上下文错误或重试裁剪，原始约束更容易完全丢失。

### 2.6 持续渗透模式会进一步泛化原始任务

在 [vulnclaw/agent/loop_controller.py](C:/Users/UncleC/Desktop/VulnClaw/vulnclaw/agent/loop_controller.py:195)：

从第 2 个 cycle 开始，传给 `auto_pentest()` 的 `user_input` 变成：

- `"[Persistent Cycle {cycle_num}] 继续对目标 ... 进行渗透测试。"`
- `"保持之前的所有发现继续深入。"`

这里没有重新带上首轮用户约束，因此 cycle 越往后，边界越容易完全消失。

## 3. 根因分析

根因不是“模型记忆不好”，而是当前任务模型缺少“约束条件的一等公民表示”。

当前系统设计中：

1. `target` 是结构化状态，有专门字段。
2. `phase` 是结构化状态，有专门字段。
3. `findings`、`notes`、`recon_data` 是结构化状态。
4. 但“约束条件/任务边界/scope”不是结构化状态。

这导致约束只能依赖：

- 首轮 user message
- 被压缩后的对话摘要
- 模型临时记忆

这三者都不具备硬约束语义，因此 issue 会稳定复现。

---

## 4. 影响范围

受影响的不只是“指定端口”。

以下约束类型都会有同类风险：

1. 端口约束：如“只允许测试 443 / 8080”
2. 路径约束：如“只测 `/admin`，不要扫全站”
3. 协议约束：如“只测 HTTPS，不要碰 SSH”
4. 动作约束：如“只做信息收集，不做利用”
5. 资产范围约束：如“只测 `api.example.com`，不要碰其他子域名”
6. 凭据/身份约束：如“仅使用提供的低权限账号，不允许爆破”
7. 输出约束：如“只生成验证结果，不要生成 PoC”

其中最危险的是：

- 范围型约束被忘记，导致越权测试
- 动作型约束被忘记，导致用户只想 recon 却执行 exploit

## 5. 开发建议

### 5.1 目标

把“任务约束条件”从自然语言上下文，升级为 Agent 运行时的结构化硬约束。

### 5.2 建议方案

#### A. 新增结构化约束模型

建议在 `SessionState` 或独立任务状态中新增：

- `task_constraints: TaskConstraints`

建议字段：

- `allowed_ports: list[int]`
- `blocked_ports: list[int]`
- `allowed_hosts: list[str]`
- `blocked_hosts: list[str]`
- `allowed_paths: list[str]`
- `blocked_paths: list[str]`
- `allowed_actions: list[str]`
- `blocked_actions: list[str]`
- `notes: list[str]`
- `strict_mode: bool`

#### B. 首轮输入解析约束

在任务启动时，从用户输入中抽取约束并写入结构体。

例如：

- “只测 443 端口” -> `allowed_ports=[443]`
- “不要碰 22 和 3306” -> `blocked_ports=[22,3306]`
- “仅做信息收集” -> `allowed_actions=['recon']`
- “不要利用漏洞” -> `blocked_actions=['exploit']`

第一阶段不必追求 NLP 全自动完美解析，可以先支持明确模式和 CLI 参数。

#### C. 每轮 prompt 强制注入约束

在 `build_dynamic_system_prompt()` 或 `build_round_context()` 中新增固定区块：

`## 当前任务硬约束`

例如：

- 仅允许测试端口：443
- 禁止测试端口：22, 3306
- 仅允许动作：recon, vuln_discovery
- 禁止动作：exploit
- 若发现新的资产/端口不在范围内，只记录，不主动测试

这个区块必须每轮都出现，不能像 `user_vuln_hint_rounds` 一样递减。

#### D. 工具调用前做硬拦截

仅靠 prompt 不够，建议在工具执行层做代码级校验。

例如：

- `nmap_scan` 执行前检查 `ports`
- `fetch` / Web 工具调用前检查目标端口、host、path 是否越界
- 如果超出约束，直接返回结构化拒绝：
  - `error_type = "constraint_violation"`
  - `message = "Port 80 is outside allowed scope [443]"`

这样即使模型忘了，执行层也不会越界。

#### E. 持续渗透模式保留原始约束

`persistent_pentest()` 在 cycle > 1 时，不应只用泛化提示。

建议：

1. 保留首轮结构化约束状态
2. cycle 提示中明确回放硬约束摘要
3. 在 resume / snapshot 恢复时同步恢复约束字段

#### F. 会话压缩时保留约束摘要

`_compress_messages()` 和 `trim_messages()` 不应该让任务边界丢失。

建议把约束从 message history 中分离出来，独立存储；同时压缩摘要里固定附带：

- 当前范围限制
- 当前禁止动作
- 当前允许资产

## 6. 验收标准

### 功能验收

1. 用户指定“只测 443 端口”后，任意 round 都不能主动测试其他端口。
2. 持续渗透跨 cycle 后，约束仍保持有效。
3. target state snapshot / resume 后，约束仍保持有效。
4. 约束冲突时，工具层返回明确的 `constraint_violation`，而不是继续执行。

### 测试验收

建议新增测试：

1. `auto_pentest` 首轮带 `allowed_ports=[443]`，第 5 轮 round context 仍包含该约束。
2. `persistent_pentest` 第 2 个 cycle 的 prompt 仍包含首轮约束。
3. `nmap_scan` 请求 `ports=80` 且 `allowed_ports=[443]` 时被拒绝。
4. `fetch` 请求 `https://target:8443/` 且只允许 `443` 时被拒绝。
5. 历史压缩后，约束字段仍可从状态中读取，不依赖旧 message。

## 9. 总结

这个 issue 的本质是：

当前 VulnClaw 已经把 `target`、`phase`、`findings` 做成了结构化状态，但没有把“任务约束”做成结构化状态。

所以约束只能依赖对话记忆，而对话记忆天然会被压缩、裁剪和泛化，导致随着 round/cycle 增加约束逐步衰减，最终越出边界。

因此，正确修复方向不是“再提醒模型一下”，而是：

- 让约束进入状态模型
- 让约束每轮强制注入
- 让工具执行层硬校验

只有这样，约束条件才不会在长链路自主渗透中失真。
